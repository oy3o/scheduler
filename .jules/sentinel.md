## 2024-05-15 - Prevent Stack Trace Leakage in Public Asynchronous APIs
**Vulnerability:** The `future.go` returned panic execution errors directly to the caller via `Future.Get()`, which included the full internal stack trace of the worker node. Furthermore, `Gatekeeper.safeOnError` logged task timeouts directly to the standard error stream (`os.Stderr`) instead of exclusively using the provided `OnError` telemetry hook.
**Learning:** Returning a raw unhandled exception or panic stack trace from a library exposes internal directory structures and dependency paths. Additionally, standard library logging (like `fmt.Fprintf(os.Stderr)`) inside a reusable library breaches the caller's operational control, risking unmonitored log output containing potentially sensitive internal state data.
**Prevention:** Always separate public-facing error surfaces from internal telemetry surfaces. Create a sanitized error payload for API boundaries (e.g., `fmt.Errorf("task panicked: %v", p)` for the public caller), while preserving the full stack trace for internal hooks designed for telemetry (e.g. `Gatekeeper.safeOnError`). Never use hardcoded output streams like `os.Stderr` in reusable libraries; instead, utilize provided hooks, or drop errors silently if no appropriate hook is configured.
